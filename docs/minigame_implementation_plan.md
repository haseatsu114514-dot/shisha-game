# シーシャ作りミニゲーム 実装計画書

> 作成日: 2026-03-08
> 対象: 大会パート（`scripts/tournament/ch*_tournament.gd`）

---

## 概要

シーシャを作る工程をミニゲームとしてゲームに落とし込む。
「シーシャ好きが触って嬉しいリアリティ」と「ゲームとして成立する面白さ」の両立を目指す。

### 変更サマリー

| 工程 | 現状 | 変更内容 |
|---|---|---|
| パッキング盛り方 | なし（配合スライダーのみ） | **新規追加**: 3択選択 + アニメーション |
| アルミ穴あけ | リズムゲーム（不安定） | **作り直し**: タップ式穴あけゲームへ変更 |
| 炭焼き | 3択（早い/ちょうど/遅い） | **拡張**: 炭の色を見て焼き加減を判定するゲームへ |
| 水量 | なし | **見送り**（後述） |
| 調整フェーズ | ゲージ止め + メニュー | **改修**: 灰落としアクション追加、数値表示の撤廃 |
| プレゼン | 軸を選ぶだけ | **大幅改修**: 自分の工程を振り返る「手札プレゼン」方式へ |

弾幕シューティング・吸い出しゲージは現状で安定しているため変更しない。

## 1. 盛り方選択（新規追加）

### コンセプト

パッキング配合（スライダー）の直後に挿入する新ステップ。
フレーバーをボウルにどう詰めるかを3択で選び、主人公が詰めるアニメーションを再生する。

### ゲーム仕様

**操作**: 3つのボタンから1つを選択

| 選択肢 | 説明テキスト | ゲーム内効果 |
|---|---|---|
| ふわふわ（エアリー） | 空気を含ませてふんわり盛る。繊細な味が出やすい | 煙量-、味のクリアさ+、後半やや不安定 |
| ふつう（セミデンス） | 適度に詰めてバランスを取る。安定した仕上がり | 全体的にバランス型、ペナルティなし |
| しっかり（オーバーパック） | ぎゅっと詰め込む。煙量が出るが焦げやすい | 煙量+、温度管理がシビアに（蒸らし最適値が狭まる） |

**アニメーション演出**:
- 選択後、テキストウィンドウに主人公の独白が入る
  - ふわふわ: 「空気を含ませるように……ふんわりと盛っていく」
  - ふつう: 「いつも通り、しっかり詰める」
  - しっかり: 「ぎゅっ……と。隙間なく詰め込んでいく」
- `_BowlVisual` のフレーバー層の描画密度を選択に応じて変える（線の間隔で表現）

**後続ステップへの影響**:

```
# heat_state への影響
ふわふわ   → _heat_state -= 1  (熱が通りやすい＝低めで十分)
ふつう     → 変化なし
しっかり   → _heat_state += 1  (熱が必要＝高めが最適)

# 蒸らし最適レンジへの影響
ふわふわ   → 最適レンジが短め寄り (5-7分)
しっかり   → 最適レンジが長め寄り (7-9分) かつ幅が狭い

# 吸い出しゲージへの影響
ふわふわ   → ゲージ速度 +5%（軽いドロー）
しっかり   → ゲージ速度 -5%（重いドロー）

# プレゼン「味」スコアへの影響
ふわふわ   → 味スコア +5
しっかり   → 煙量スコア +5
```

### 実装方針

- `_show_packing_step()` の完了後、新関数 `_show_packing_style_step()` を呼ぶ
- 既存の `_add_choice_button()` で3ボタンを生成
- 内部変数 `_packing_style: String` を追加（"airy" / "normal" / "dense"）
- 後続ステップで `_packing_style` を参照して各種パラメータを調整
- アニメーションは独白テキスト + `_BowlVisual` の再描画で表現

### 工数目安

選択肢UI + パラメータ反映 + テキスト追加のみ。既存パターンの踏襲で済む。
**最も実装が軽い変更。**

## 2. アルミ穴あけ（作り直し）

### 現状の問題

現在のリズムゲーム（円形リング上のノーツをタイミングで叩く）は何度も破綻している。
リズムゲームとしての面白さと、「穴を開けている」実感が両立しにくい構造が根本原因。

### 新コンセプト: タップ式穴あけゲーム

アルミホイルの上面図を表示し、光るターゲットをタップして穴を開けていく。
「正確に穴を開ける」という実際の作業感を直接再現する。

### ゲーム仕様

**画面構成**:
```
┌──────────────────────────┐
│                          │
│     ○ ○ ○ ○ ○ ○         │  ← アルミホイル上面図（円形）
│    ○ ○ ○ ○ ○ ○ ○        │
│   ○ ○ ○ ● ○ ○ ○ ○       │  ● = 現在光っているターゲット
│    ○ ○ ○ ○ ○ ○ ○        │  ★ = 開け済みの穴
│     ○ ○ ○ ○ ○ ○         │
│                          │
│   [残り: 18/24]          │
│   [精度: ★★★]            │
└──────────────────────────┘
```

**操作フロー**:
1. アルミホイルの上に穴位置が薄いガイドとして表示される（24〜30穴程度）
2. ガイド穴が1つずつ（または2〜3個同時に）光る
3. 光っている間にタップ/クリックすると穴が開く
4. タップのタイミングで精度が決まる:
   - 光り始めてすぐ → PERFECT（穴がきれい）
   - 光の後半 → GOOD（まあまあ）
   - 光が消えかけ → NEAR（ずれた穴）
   - 光が消えた後 or 間違った場所 → MISS
5. 全ガイド穴を処理したら終了

**テンポ**:
- 1穴あたり約0.8〜1.2秒の猶予（リズムゲームより余裕がある）
- 穴が順番に光るので、プレイヤーは「次はどこだ」と画面を見る必要がある
- 後半にかけて少しテンポが上がる（焦りの演出）

**ゲーム的な面白さ**:
- リズムゲームの「タイミングを合わせる快感」は維持
- 「どこに穴が開いているか」が視覚的にわかるので、穴あけの実感がある
- 完了後、開けた穴のパターンがそのまま結果ビジュアルとして残る

**判定・スコアリング**:

```gdscript
# 精度スコア計算
var accuracy = (perfect_count * 1.0 + good_count * 0.7 + near_count * 0.3) / total_holes

# 技術ポイントへの反映
var delta_spec = lerp(-8.0, 18.0, accuracy)

# ゾーンボーナスへの反映（穴の均一さが熱分布に影響）
_zone_bonus += lerp(-0.1, 0.2, accuracy)
```

**難易度調整**:
- イージーモード: 光る時間が長い（1.5秒）、同時に光る穴は1つだけ
- ノーマル: 光る時間は1.0秒、後半は2穴同時
- 機材による変化: HMS によって穴の総数が変わる（直炭なら多い、HMSなら少なめ）

### 実装方針

**削除するもの**:
- `_AluminumRingVisual` クラス（円形リングビジュアル）
- `_on_aluminum_tick()` のノーツ移動ロジック
- `_spawn_aluminum_note()` のノーツ生成ロジック

**新規作成するもの**:

```
_AluminumFoilVisual (Control)
├── 円形のアルミホイル背景を描画
├── ガイド穴を薄い円で描画（位置は事前計算）
├── 光っている穴をハイライト描画
└── 開け済みの穴を塗りつぶし描画

_show_aluminum_step()  ← 大幅書き換え
├── _AluminumFoilVisual を生成・配置
├── タップボタンの代わりに、ビジュアル上の穴をクリック可能にする
│   （またはシンプルに1ボタン「穴を開ける！」を維持）
├── タイマーで穴を順次ハイライト
└── 全穴処理後にスコア計算 → 次ステップへ
```

**操作方式の選択**:

穴を直接クリックする方式は座標判定が複雑になるため、
**既存の「穴を開ける！」ボタンを維持**し、
光っている穴があるタイミングで押す方式を採用する。
（ボタン1つで操作が完結するので、キーボード操作との互換性も保てる）

つまり構造としては:
- ビジュアルが「円形リング → アルミホイル上面図」に変わる
- 操作は「光ったらボタンを押す」で基本同じ
- ノーツが移動する代わりに、穴が順番に光る
- 精度判定は「光ってからボタンを押すまでの時間差」で行う

### 旧リズムゲームとの違いまとめ

| 項目 | 旧（リズムゲーム） | 新（タップ式） |
|---|---|---|
| ビジュアル | 円形リング上のノーツが移動 | アルミホイル上面に穴が光る |
| 操作 | ノーツがヒット位置に来たら押す | 穴が光ったら押す |
| 判定 | 移動するノーツとの距離 | 光ってからの反応時間 |
| テンポ | 一定間隔（リズム依存） | 穴ごとに光る（可変間隔可） |
| 実感 | 「リズムを刻む」 | 「穴を1つずつ開ける」 |
| 安定性 | ノーツの移動・衝突判定が複雑 | 状態管理がシンプル |

## 3. 炭焼き判定（拡張）

### 現状の問題

「早い/ちょうど/遅い」の3択は、炭を焼く体験としては物足りない。
一方、実際の炭焼きは10分かかるので、リアルタイムに焼く演出は違和感がある。

### 新コンセプト: 時間経過後の「焼き加減判定」

**「炭をコンロにセットして、しばらく待つ」演出を挟んだ上で、焼き加減を目視で判断する。**

実際のシーシャ作りでも、炭を焼いている間は他の準備（ボウルのパッキングなど）をしている。
この「並行作業」感を活かし、パッキング等の作業中にバックグラウンドで炭が焼けていく構成にする。

### ゲーム仕様

**フロー**:

```
[セッティング]
  ↓
「炭をコンロにセット」 ← ここで炭タイマー開始（画面端に小さく表示）
  ↓
[フレーバー選択]       ← 炭がバックグラウンドで焼けていく
  ↓
[パッキング配合]       ← 同上
  ↓
[盛り方選択]           ← 同上
  ↓
[アルミ穴あけ]         ← 同上
  ↓
「炭の様子を確認」     ← ★ここで炭焼き判定ミニゲーム
  ↓
[炭の配置]
```

**炭タイマー演出**:
- セッティング完了時に画面右上に小さな炭アイコンが表示される
- 各ステップを進めるごとに炭の色が黒→赤みがかっていく
- プレイヤーが各ステップに費やした時間も内部で記録（早くスキップ=炭がまだ、じっくり=よく焼ける）

**判定ゲーム本体**:

```
┌──────────────────────────────┐
│                              │
│    🟫🟫      🟫🟫            │
│    🟥🟫      🟥🟥            │  ← 3個の炭が並んでいる
│    🟥🟥      🟫🟥            │    色は現在の焼け具合を反映
│                              │
│  この焼き加減でいく？          │
│                              │
│  [ もう少し待つ ]              │  ← 待つと色がさらに変わる
│  [ ちょうどいい！ ]            │  ← 今の状態で確定
│  [ 裏返す ]                   │  ← 裏返して追加加熱
│                              │
└──────────────────────────────┘
```

**操作フロー**:
1. 炭のビジュアルが表示される。色は焼け具合を反映している
2. プレイヤーは3つの行動から選ぶ:
   - **「もう少し待つ」**: 1段階焼きが進む（最大3回まで押せる）
   - **「ちょうどいい！」**: 現在の焼け具合で確定する
   - **「裏返す」**: 炭を裏返す。裏面がまだ焼けていない場合に有効
3. 炭の状態は5段階:

| 段階 | 見た目 | 判定 |
|---|---|---|
| 1 | ほぼ黒、端だけ赤い | 生焼け（NG） |
| 2 | 半分赤い | やや早い |
| 3 | 全体が赤く、角まで火が回っている | **ベスト** |
| 4 | 赤が強く、表面に白い灰が見え始める | やや焼きすぎ |
| 5 | 灰が多い、赤みが薄れている | 焼きすぎ（NG） |

**ゲーム的な面白さ**:
- 「もう少し待つ」を押すたびにビジュアルが変わるので、**変化を見て判断する楽しさ**がある
- 裏返しのタイミングも判断要素（片面だけ焼けすぎると使いにくい炭になる）
- 前のステップをゆっくりやっていた場合、すでに段階3〜4で始まるので「急いで確定しなきゃ」という緊張感も
- バックグラウンドタイマーの存在で「他の作業中も炭を気にしている」感覚が出る

**スコアリング**:

```gdscript
# 焼き加減によるスコア
match charcoal_stage:
    1: delta_spec = -8.0; _heat_state -= 2   # 生焼け
    2: delta_spec = 3.0;  _heat_state -= 1   # やや早い
    3: delta_spec = 12.0; _heat_state += 0    # ベスト
    4: delta_spec = 5.0;  _heat_state += 1    # やや焼きすぎ
    5: delta_spec = -5.0; _heat_state += 2    # 焼きすぎ

# 裏表の焼きムラペナルティ
if front_stage != back_stage and abs(front_stage - back_stage) >= 2:
    delta_spec -= 4.0  # 片面焼けすぎ
```

### 実装方針

- `_show_charcoal_prep_step()` を大幅書き換え
- 新しい内部変数:
  - `_charcoal_stage: int` (1-5)
  - `_charcoal_flipped: bool`
  - `_charcoal_back_stage: int` (裏面の焼き段階)
  - `_charcoal_bg_timer: float` (バックグラウンド経過時間)
- バックグラウンドタイマーは `_process()` で毎フレーム更新するのではなく、
  各ステップ完了時に `_charcoal_bg_timer += step_elapsed_time` で加算する（シンプル）
- ビジュアルは `_CharcoalVisual` カスタムControlクラスを新規作成
  - `_draw()` で炭を3〜4個描画（`draw_rect` + グラデーション色）
  - 焼き段階に応じて色を `Color.lerp()` で補間:
    - 黒 `#2a2a2a` → 赤 `#cc3300` → 白灰 `#999999`

### 工数目安

バックグラウンドタイマー + ビジュアル + 判定ロジックの3要素。
ビジュアルは `_BowlVisual` と同程度の `_draw()` 実装。中規模の変更。

## 4. 調整フェーズ（改修）

### 現状の問題

1. **同じゲージゲームの繰り返しで飽きる**: 方向を選ぶ → ゲージを止める、の2ステップを最低2回。やることが毎回同じ
2. **数値が丸見え**: `熱状態: 2` のように内部数値がそのまま表示されている。CLAUDE.mdの「数値は絶対にプレイヤーに見せない」ルールに違反
3. **行動の手応えがない**: 炭を増減しても「調整した」というテキストが出るだけ。何がどう変わったかの実感がない
4. **「最低2回アクション」が人工的**: なぜ2回なのかの理由がなく、義務感だけが残る

### 新コンセプト: 「炭の世話」アクションメニュー

調整フェーズを「シーシャの面倒を見る時間」として再設計する。
メニュー選択の構造は維持しつつ、行動の種類を増やし、ビジュアルフィードバックを強化する。

### ゲーム仕様

**画面構成**:

```
┌──────────────────────────────────────────┐
│  ラウンド2: 調整タイム                     │
│                                          │
│  ┌────────────────────┐                  │
│  │  [ボウル上面図]      │  煙の様子:       │
│  │   炭 ● ●            │  もくもくと白い煙 │
│  │      ●              │                  │
│  │  HMS: 半開き         │  審査員の反応:     │
│  └────────────────────┘  南雲が少し         │
│                          眉をひそめた。     │
│  ┌─────────────────────────────────────┐  │
│  │ 🔥 灰を落とす                        │  │
│  │ ♻️ 炭の位置を回す                     │  │
│  │ 🔄 炭を交換する                       │  │
│  │ 🎚️ HMSのフタを調整する                │  │
│  │ ✅ 調整を終える                        │  │
│  └─────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

**行動メニュー（5種類）**:

| 行動 | 説明 | 効果 | リアリティ |
|---|---|---|---|
| 灰を落とす | 炭の灰をトントンと落とす | 熱効率が回復。_heat_stateが0に1段階近づく | シーシャ屋で実際にやる最頻アクション |
| 炭の位置を回す | HMS上の炭を90度回転させる | ボウルの焼けムラを解消。次ラウンドの安定度UP | 実際の調整で非常に重要 |
| 炭を交換する | 古い炭を新しい炭に入れ替える | 熱量がリセットされる。_heat_stateが+1寄りに | 長いセッションでは必須 |
| HMSのフタを調整する | フタの開き具合を変える | 熱の入り方を微調整。開ける=冷ます、閉める=温める | HMS使用時の基本操作 |
| 調整を終える | 次のラウンドへ進む | - | - |

**最低アクション数の撤廃**:
- 「最低2回」の制限を撤廃する
- 代わりに、何もせず「調整を終える」を選んだ場合は炭の状態が自然に悪化する（灰が溜まる、温度が偏る）
- つまり「やらないと損」だが「やらなきゃ進めない」ではない。自然な動機付け

**数値表示の撤廃**:
- `熱状態: 2` のような数値は一切表示しない
- 代わりに以下のビジュアル・テキスト表現を使う:

| 内部値 | 煙の様子（テキスト） | ボウルビジュアル |
|---|---|---|
| -3 | 煙がほとんど見えない… | 炭の色が暗い |
| -2 | 煙が細く、頼りない | 炭がやや暗い |
| -1 | 煙は出ているが、少し薄い | やや暗め |
| 0 | もくもくと白い煙が立ち上る | 炭が均一に赤い（理想） |
| +1 | 煙が濃い。少しピリッとする | 炭が明るめ |
| +2 | 煙に嫌な苦味が混じり始めた | 炭が強く赤い |
| +3 | 焦げた匂いがする。まずい。 | 炭が白灰まみれで過熱 |

**審査員リアクション**:
- 調整フェーズ中、審査員の短い反応が表示される（現状の `_build_adjustment_cue` を拡張）
- 各行動後に審査員が反応を見せる:
  - 灰を落とした → 「南雲が小さく頷いた」
  - 的外れな調整 → 「前園が首をかしげた」
- これが間接的なヒントになる（数値を見せずに状態を伝える手段）

**灰落としの小アクション**（オプション）:

灰落としだけは、単なるボタン押しではなく小さなアクションにできる。

```
「灰を落とす」を選択
  ↓
炭のビジュアルが拡大表示される
  ↓
タイミングよくボタンを押すと「トン！」と灰が落ちる
  ↓
1〜2回のタップで完了（3秒程度の超短アクション）
```

- 成功: 灰がきれいに落ちて炭が復活
- 失敗（押しすぎ）: 炭がHMSの外にずれる → 温度が偏る
- これにより「ただメニューを選ぶだけ」ではなく、小さな手応えが生まれる

### 実装方針

- `_show_adjustment_menu()` を書き換え、メニュー項目を5種類に変更
- `_adjustment_action_count >= 2` のゲート条件を撤廃
- `info_label` への熱状態数値表示を、煙の様子テキストに置換
- `_build_adjustment_cue()` を拡張し、行動後の審査員リアクションを追加
- 灰落としゲージは吸い出しの `_build_gauge_bar` を流用可能（既存資産の再利用）
- HMSフタ調整は新規追加だが、内部的には `_heat_state` の上下だけなのでシンプル

### 既存の吸い出し微調整ゲージについて

現在の「方向を選ぶ → ゲージを止める」方式は削除する。
代わりに上記の5種類のアクションで調整を行う。ゲージゲームは吸い出し本編で十分に登場するため、調整フェーズでの重複を避ける。

## 5. プレゼン（大幅改修）

### 現状の問題

1. **「軸を選ぶだけ」で体験が薄い**: 味/煙/吸いやすさ/個性から1〜2個選んで終わり。「プレゼンしている感」がゼロ
2. **自分の工程との繋がりが弱い**: ふわふわパッキングにした理由も、炭4個にした意図も、プレゼンで語る機会がない
3. **審査員とのやりとりがない**: 一方的に選ぶだけで、審査員の反応や質問がない
4. **スコアの適性値が不透明**: 「適性 68」と表示されても、なぜ68なのかプレイヤーにはわからない

### 新コンセプト: 「手札プレゼン」方式

**大会中に自分がやった工程が「プレゼンカード（手札）」になり、それを選んで審査員に語る。**

ダンガンロンパの「言弾（コトダマ）」のように、自分のプレイ履歴がそのままプレゼンの武器になる仕組み。

### ゲーム仕様

**手札の生成ルール**:

大会の各工程で行った選択が、プレゼン時に自動的に「手札」として手元に並ぶ。

```
例: プレイヤーが以下の選択をした場合

フレーバー: ダブルアップル + ミント
盛り方: ふわふわ
穴あけ: 精度GOOD以上
炭: 3個、ベストな焼き加減
蒸らし: 7分
吸い出し: 3回で完成
調整: 灰落とし成功

  ↓ 以下の手札が生成される

[ダブルアップルの深み]   ← フレーバー由来
[ミントの清涼感]        ← フレーバー由来
[エアリーな口当たり]    ← 盛り方由来
[丁寧な穴あけ]          ← 穴あけ精度由来
[安定の3個炭]           ← 炭配置由来
[じっくり7分蒸らし]     ← 蒸らし時間由来
[3回の吸い出しで仕上げ] ← 吸い出し由来
```

**手札の属性**:

各手札には、4軸（味/煙/吸いやすさ/個性）へのスコア影響がついている。

```
[エアリーな口当たり]
  味 +8 / 煙 -2 / 吸いやすさ +5 / 個性 +3

[安定の3個炭]
  味 +3 / 煙 +2 / 吸いやすさ +6 / 個性 -1
```

ただし、この数値はプレイヤーには見せない。
代わりに、手札の色合いやアイコンで「味寄りの手札」「煙寄りの手札」が直感的にわかるようにする。

**プレゼンのフロー**:

```
Phase 1: 審査員が質問する
  ↓
Phase 2: 手札から1枚選んで答える（3ターン）
  ↓
Phase 3: 結果発表
```

**Phase 1 - 審査員の質問**:

審査員が3回質問する。質問は審査員の好みに基づく。

```
南雲「……味の設計について聞かせてくれ」        → 味系の手札が刺さる
前園「煙の量はどう考えた？おいしそうだねえ」   → 煙系の手札が刺さる
エミル「なぜこの作り方を選んだんだい？」       → 個性系の手札が刺さる
```

**Phase 2 - 手札を出す**:

質問に対して、手元の手札から1枚を選んで出す。

```
┌──────────────────────────────────────────┐
│  南雲「味の設計について聞かせてくれ」       │
│                                          │
│  手札を1枚選んでください:                  │
│                                          │
│  ┌──────────────┐ ┌──────────────┐       │
│  │🍎 ダブル      │ │🌿 ミントの   │       │
│  │ アップルの深み │ │  清涼感      │       │
│  │              │ │              │       │
│  └──────────────┘ └──────────────┘       │
│  ┌──────────────┐ ┌──────────────┐       │
│  │☁️ エアリーな  │ │🔥 安定の     │       │
│  │  口当たり     │ │  3個炭       │       │
│  └──────────────┘ └──────────────┘       │
│  ... (他の手札)                           │
└──────────────────────────────────────────┘
```

手札を選ぶと、主人公がその手札に基づいたセリフを自動生成して話す:

```
はじめ「ダブルアップルの甘みを活かすために、
       エアリーなパッキングで香りの通りを良くしました」
```

**審査員リアクション**:

手札と質問の相性で、審査員のリアクションが変わる:

- **大正解（質問の軸に強い手札）**: 南雲が目を見開いた。「……なるほど」
- **まあまあ（関連はある）**: 南雲が小さく頷いた。
- **的外れ（質問と無関係な手札）**: 南雲の表情は変わらない。「……それで？」
- **矛盾（やったことと言ったことが矛盾）**: 南雲が眉をひそめた。「しかし実際の煙は……」

**矛盾システム**:

「ふわふわパッキングにしたのに煙量を自慢する」など、
自分のプレイ内容と矛盾するプレゼンをすると減点される。
逆に、一貫性のあるプレゼン（エアリー → 味重視 → 繊細さアピール）はボーナス。

```gdscript
# 一貫性ボーナスの判定
# 出した手札3枚が同じ軸に偏っている場合
var dominant_axis = _get_dominant_axis(played_cards)
if dominant_axis != "":
    # 全手札が同じ方向を向いている → ストーリーとして一貫している
    consistency_bonus = 6.0
    lines.append("プレゼン全体に一貫性があり、説得力が増した。")
```

**使用済み手札**:

一度出した手札は使えない。3ターンで3枚を選ぶ戦略性がある。
「最初に強い手札を出すか、質問に合わせて温存するか」の判断。

- `_resolve_presentation_focus()` と関連関数を全て書き換え
- 新規関数群:

  - `_show_presentation_question(turn: int)`: 審査員の質問を表示

  - `_resolve_presentation_turn(card: Dictionary, turn: int)`: 1ターンの結果計算

- 審査員の質問は `JUDGE_FOCUS_PREFERENCES` を流用（既存データ活用）
- セリフテンプレートは定数として定義（dialogue JSONに入れるほどの量ではない）

### 工数目安

手札生成 + 質問ターン制UI + セリフテンプレート + スコアリング。
ロジックは複雑だがUI構造は既存の選択肢パターンの延長。中〜大規模の変更。

## 6. 水量調整について（見送り）

### 見送り理由

- ゲージを止めるタイプのゲームは「吸い出し」と操作感がかぶる
- 大会シーンではセッティング工程が既に多く、テンポが悪くなるリスク
- 水量は「選択」で十分に表現可能

### 代替案（将来検討）

日常パート（練習モード）で水量調整を体験できるようにする。
大会パートでは、機材選択時に水量の「多め/標準/少なめ」を選択肢として組み込む。
これならステップ増加なしで水量の概念を取り入れられる。

## 7. フレーバー特性システム（新規追加）

### コンセプト

フレーバーの `category`（cooling / spice / fruit / sweet / floral / tea）をゲームプレイに反映させる。
シーシャ未経験者でも「フレーバーによって作り方が変わるんだ」と自然に理解できる設計にする。

### カテゴリ別の特性パラメータ

`data/flavors.json` の各フレーバーに以下のプロパティを追加する:

```json
{
  "id": "mint",
  "name": "アルファーヘブン ミント",
  "category": "cooling",
  "heat_tolerance": 1.2,
  "smoke_weight": 0.7,
  "steam_bias": -1,
  "price": 1550,
  "description": "..."
}
```

| カテゴリ | 代表例 | heat_tolerance | smoke_weight | steam_bias | 理由 |
|---|---|---|---|---|---|
| cooling | ミント | 1.2（高い） | 0.7（軽い） | -1（短め寄り） | ミントは熱に強く、高温でもキレが出る |
| spice | ダブルアップル | 1.1（高い） | 1.3（重い） | +1（長め寄り） | アニス系は熱でじわじわ開く |
| fruit | ブルーベリー, パイナップル | 0.7（低い） | 0.8（軽い） | -1（短め寄り） | フルーツ系は繊細で焦げやすい |
| sweet | バニラ, ココナッツ | 1.0（中間） | 1.0（中間） | +1（長め寄り） | じっくり蒸らすと甘みの厚みが出る |
| floral | ローズ, ラベンダー | 0.8（やや低い） | 0.9（やや軽い） | 0（標準） | 繊細だが極端に弱くはない |
| tea | アールグレイ | 0.9（やや低い） | 1.0（中間） | 0（標準） | 渋みが出やすく温度管理に注意 |

### ミックスの特性値計算

複数フレーバーを選んだ場合、**グラム数による加重平均**で全体のパラメータを算出する:

```gdscript
func _calc_mix_properties(flavors: Array[String], grams: Dictionary) -> Dictionary:
    var total_grams := 0.0
    var heat_tol := 0.0
    var smoke_w := 0.0
    var steam_b := 0.0
    for fid in flavors:
        var g = float(grams.get(fid, 0))
        var fdata = _get_flavor_data(fid)
        heat_tol += fdata.heat_tolerance * g
        smoke_w  += fdata.smoke_weight * g
        steam_b  += fdata.steam_bias * g
        total_grams += g
    if total_grams > 0:
        heat_tol /= total_grams
        smoke_w  /= total_grams
        steam_b  /= total_grams
    return {"heat_tolerance": heat_tol, "smoke_weight": smoke_w, "steam_bias": steam_b}
```

### 後続工程への影響

**1. 蒸らし最適レンジがフレーバーで変わる**

```
基本最適レンジ: 6-8分
steam_bias の影響:
  -1（cooling/fruit主体） → 5-7分が最適
  +1（spice/sweet主体）  → 7-9分が最適
  盛り方との複合: ふわふわ + fruit → 最適が4-6分まで短縮される場合も
```

**2. 温度管理（heat_state）の許容幅が変わる**

```
heat_tolerance の影響:
  1.2（ミント主体） → heat_state ±3 まで減点なし（余裕がある）
  0.7（フルーツ主体）→ heat_state ±1 で減点開始（シビア）
  具体的には _heat_penalty_threshold を heat_tolerance で補正
```

**3. 吸い出しゲージ速度がフレーバーで変わる**

```
smoke_weight の影響:
  0.7（軽い煙）→ ゲージ速度 +10%（ドローが軽い）
  1.3（重い煙）→ ゲージ速度 -10%（ドローが重い）
  盛り方しっかり + ダブルアップル → かなり重いドローになる
```

**4. プレゼン手札の内容が変わる**

フレーバーカテゴリに応じて、プレゼン手札の名前と属性値が変わる:

| カテゴリ | 手札名の例 | 味 | 煙 | 吸いやすさ | 個性 |
|---|---|---|---|---|---|

### ミックス相性の設計方針（確定）

実際のシーシャでは大体何を混ぜてもそこそこ美味しくなる。
この現実に合わせ、**カテゴリ単位の自動相性判定は廃止**する。

代わりに、以下の3層構造で味スコアを決定する:

```
1. デフォルト = 普通に美味しい（ボーナスなし・ペナルティなし）
2. レシピ成立 = めちゃくちゃ良い（ボーナス付き）
3. NGミックス = 本当にひどい（ペナルティ付き）
```

**デフォルト（大多数の組み合わせ）**:
- 何を混ぜても味スコアに補正なし
- フレーバー特性（heat_tolerance / smoke_weight / steam_bias）は工程への影響として残る
- つまりフレーバー選択は「味の相性」ではなく「作り方の変化」に影響する

**レシピ成立（30〜50個）**:
- セクション8の特別レシピ発見システムで管理
- 日常パートで入手したレシピの組み合わせ＋比率を満たすとボーナス
- 「良い組み合わせ」の表現はすべてレシピに集約する

**NGミックス（5〜10個を手動定義）**:
- 本当にまずい・ありえない組み合わせだけをハードコードで定義
- `data/ng_mixes.json` に個別フレーバーの組み合わせで記載
- 例: （具体的なNGミックスはフレーバーラインナップ確定後に手動で選定）

```json
{
  "ng_mixes": [
    {
      "flavors": ["flavor_a", "flavor_b"],
      "penalty": -5.0,
      "text": "この組み合わせは味が喧嘩してしまった。"
    }
  ]
}
```

この方式の利点:
- 55種×N種の全組み合わせを定義する必要がない
- 実際のシーシャの「大体何混ぜても美味しい」感覚に忠実
- プレイヤーが自由にフレーバーを選んでも不当に罰されない
- 「良い組み合わせ」の発見はレシピシステムの楽しさに統合される

### ratio_tolerance（比率の許容幅）

特別ミックスの判定を厳密な比率ではなく、ある程度の幅を持たせる:

```gdscript
func _detect_recipe_match(selected_flavors: Array, grams: Dictionary) -> Dictionary:
    for recipe in _known_recipes:
        var recipe_flavors = recipe.get("flavors", [])
        # フレーバーの種類が一致するか
        if not _arrays_match_unordered(selected_flavors, recipe_flavors):
            continue
        # 比率の判定（tolerance 分のずれを許容）
        var target_ratio = recipe.get("ratio", [])
        var tolerance = recipe.get("ratio_tolerance", 1)
        var actual = []
        for fid in recipe_flavors:
            actual.append(int(grams.get(fid, 0)))
        if _ratio_matches(actual, target_ratio, tolerance):
            return recipe
    return {}
```

- `ratio_tolerance: 1` → 各フレーバーが ±1g の範囲ならOK
- `ratio_tolerance: 2` → ±2g まで許容（比率にこだわらなくても成立しやすい）
- 完全一致を求めず、「なんとなくこの組み合わせで作ったら成立した」感を出す

**レシピを知らなくても偶然同じ配合にしたら成立する**か、**レシピを覚えていないと成立しない**か:

- 偶然成立すると、プレイヤーが「なぜボーナスが入ったかわからない」状態になる


### 2. アルミ穴あけリデザインの章別デザイン

ch1 で実装した「上面グリッドタップ方式」を ch2〜ch4 に展開する。
章が進むにつれて**穴の数・タイミング・パターン**が進化する。

#### 共通変更（全章）

- `_AluminumRingVisual` を `_AluminumGridVisual` に置換
- リズムゲーム方式（notes + distance ベースの衝突判定）を完全削除
- 「光る → タップ」のグロウタイマー方式に統一
- `_aluminum_glow_timer` の追加
- `_on_aluminum_tick()` を空にし、`_on_aluminum_glow_tick()` を使用

#### ch2: 「テンポアップ」

ch1 の基本をそのまま引き継ぎつつ、**テンポが速い**。

- 成功率 75% がクリアライン（ch1 と同じ）
- DJ SMOKE が穴あけ中にヤジを飛ばす（「お、リズム悪いぞ！」等）
  → ヤジは info_label に一瞬表示して消える演出

#### ch3: 「ダブルグロウ」

ch3 から**同時に2箇所が光る**場面が登場する。

```
穴の数:     28〜32
グロウ時間:  0.9秒
ダブルグロウ確率: 25%（4穴に1回の割合で次の穴も同時に光る）
```

**ダブルグロウのルール:**
- 2箇所が同時に光っている時、**どちらをタップしても1穴分の成功判定**
- ただし**素早く2回タップして両方成功させると「DOUBLE!」判定**でボーナス
- DOUBLE 判定: spec +1.5、zone +0.02（1回あたり）
- 2つ目を取れなくても MISS にはならない（あくまでボーナス）

```gdscript
# ch3 ダブルグロウ
var _aluminum_double_glow: bool = false
var _aluminum_double_hole: int = -1
var _aluminum_double_hits: int = 0

func _start_next_aluminum_glow() -> void:
    # 25% の確率でダブルグロウ
    _aluminum_double_glow = randf() < 0.25 and _aluminum_current_hole + 1 < _aluminum_grid_holes.size()
    if _aluminum_double_glow:
        _aluminum_double_hole = _aluminum_current_hole + 1
    _aluminum_glow_active = true
    _aluminum_glow_elapsed = 0.0
    _aluminum_glow_timer.start()
```

- ビジュアル: ダブルグロウ時は2つの穴が同時に光り、接続線（破線）で繋がれる
- ナンディ/スティーブが「2穴同時は高等テクニックだ」と解説
- DOUBLE 成功時の SE は通常と異なる（「confirm」ではなく専用 SE）

#### ch4: 「パターンシーケンス」

ch4（世界大会）では穴あけに**パターン**が導入される。
光る順序がランダムではなく、**意味のある配置パターン**で出現する。

```
穴の数:     30〜36
グロウ時間:  0.8秒
パターン: 3種類がランダムに切り替わる
  - ストレート（横一列に順番に光る）
  - サークル（円を描くように光る）
  - ジグザグ（千鳥格子に光る）
ダブルグロウ確率: 35%
```

**パターンボーナス:**
- パターンが切り替わるタイミングで**0.5秒の予告表示**
  → 「パターン: ストレート」等が info_label に一瞬出る
- パターン内で**全 PERFECT** を取ると「CLEAN PATTERN!」ボーナス
  → spec +5.0, zone +0.05
- パターンは 5〜7 穴で1セット、合計5〜6パターンで30〜36穴

```gdscript
    "zigzag",     # 千鳥

var _aluminum_current_pattern: String = ""
var _aluminum_pattern_start: int = 0
var _aluminum_pattern_length: int = 6
var _aluminum_pattern_perfect_streak: bool = true

func _generate_pattern_holes() -> void:
    # パターンに基づいて穴の順序を決定
    # straight: row=固定, col=0→5
    # circle: (0,0)→(0,5)→(4,5)→(4,0)→(0,0)
    # zigzag: (0,0)→(1,1)→(0,2)→(1,3)→...
```

- Master Hookah が「パターンを読めるか」と挑発
- シェイクが「中東の伝統的な穴の配置だ」と解説
